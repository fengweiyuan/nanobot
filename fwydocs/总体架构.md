# Nanobot 总体架构

## 项目概览

**nanobot** 是一个超轻量级个人 AI 助手框架，核心代码仅 ~4000 行（比 OpenClaw 的 430k+ 行减少了 99%）。

- **目标**: 提供研究友好、易于扩展的 AI 助手基础设施
- **特色**:
  - 极简核心架构
  - 支持 20+ LLM 提供商
  - 支持 10+ 聊天平台
  - MCP (Model Context Protocol) 集成
  - 完整的工具生态（Shell、文件、网页、计划任务等）

## 核心消息处理流程

```
┌─────────────────────────────────────────────────────────────────┐
│                     Complete Message Flow                        │
└─────────────────────────────────────────────────────────────────┘

用户在 Telegram/Discord/等平台发送消息
         │
         ▼
   [Channel Adapter] 捕获消息
         │
         ▼
   构建 InboundMessage {user_id, text, channel, ...}
         │
         ▼
   [Message Bus] 入站队列
         │
         ▼
   [Agent Loop] 开始处理:
         │
         ├─→ [Session Manager] 加载对话历史
         │
         ├─→ [Context Builder] 构建 LLM 提示词:
         │   ├─ 获取对话历史（最近 N 条）
         │   ├─ 加载用户记忆（持久化事实）
         │   ├─ 加载技能（扩展功能）
         │   ├─ 注册所有可用工具
         │   └─ 生成最终系统提示词
         │
         ├─→ [LLM Provider] 调用 LLM:
         │   ├─ 通过 Provider Registry 路由
         │   ├─ 设置环境变量和 API 密钥
         │   ├─ 发送完整的 tools schema
         │   └─ 等待 LLM 响应
         │
         ├─→ 解析 LLM 输出:
         │   ├─ 提取文本内容
         │   ├─ 查找 tool_calls (JSON)
         │   ├─ 若无 tool_calls，返回最终回复
         │   └─ 若有 tool_calls，继续下一步
         │
         ├─→ [Tool Registry] 执行工具:
         │   ├─ 查找工具实现
         │   ├─ 验证参数
         │   ├─ 执行工具逻辑
         │   ├─ 捕获输出（限制 500 字符）
         │   └─ 将结果发送回 LLM
         │
         ├─→ 循环: 若 LLM 还有 tool_calls，重复上面两步
         │
         └─→ [Memory Store] 更新记忆:
             └─ 保存本次交互（用于未来上下文）

         ▼
   构建 OutboundMessage {channel, user_id, text, ...}
         │
         ▼
   [Message Bus] 出站队列
         │
         ▼
   [Channel Manager] 路由到对应渠道
         │
         ▼
   [Channel Adapter] 调用平台 API
         │
         ▼
   用户在平台上收到回复
```

## 核心模块详解

### 1. Agent Loop (`agent/loop.py`)

**职责**: 整个 AI 处理流程的调度器

```python
# 伪代码示意
async def process_message(inbound_msg):
    # 1. 加载会话
    session = load_session(inbound_msg.user_id)

    # 2. 构建上下文
    context = context_builder.build(
        history=session.history,
        memory=memory_store.load(user_id),
        skills=skills_loader.load(),
        tools=tool_registry.list_all()
    )

    # 3. LLM 循环
    while True:
        response = await llm_provider.call(context)

        if not response.tool_calls:
            break  # 无工具调用，完成

        # 4. 执行工具
        for tool_call in response.tool_calls:
            result = await tool_registry.execute(tool_call)
            context.add_tool_result(tool_call.id, result)

    # 5. 更新记忆
    memory_store.save(user_id, response.text)

    # 6. 发送回复
    return OutboundMessage(text=response.text)
```

**关键接口**:
- `process_message(inbound_msg)` - 处理入站消息
- `context_builder` - 构建提示词
- `tool_registry` - 执行工具
- `llm_provider` - 调用 LLM

### 2. Message Bus (`bus/`)

**职责**: 解耦渠道和核心处理逻辑

```python
# 架构模式: Event-Driven Queue
class MessageBus:
    """
    异步消息队列，支持:
    - InboundMessage: 用户输入（Telegram/Discord/etc）
    - OutboundMessage: 机器人回复
    """

    # 入站队列
    inbound_queue: asyncio.Queue[InboundMessage]

    # 出站队列
    outbound_queue: asyncio.Queue[OutboundMessage]

    async def dispatch_inbound(self, msg):
        """渠道将消息放入队列"""
        await self.inbound_queue.put(msg)

    async def listen_outbound(self):
        """各渠道监听输出消息"""
        while True:
            msg = await self.outbound_queue.get()
            # 路由到对应渠道
```

**优点**:
- 渠道和核心逻辑完全解耦
- 支持多渠道并发处理
- 易于添加新渠道

### 3. Channels Layer (`channels/`)

**职责**: 聊天平台的适配器

```
BaseChannel (抽象)
  ├─ TelegramChannel
  ├─ DiscordChannel
  ├─ WhatsAppChannel
  ├─ FeishuChannel
  ├─ SlackChannel
  ├─ EmailChannel
  ├─ DingTalkChannel
  ├─ MatrixChannel
  ├─ QQChannel
  └─ MockchatChannel
```

**每个渠道的职责**:
1. **接收**: 监听平台 API，捕获用户消息
2. **转换**: 转换为统一的 `InboundMessage` 格式
3. **放入队列**: 将消息发送到 Message Bus
4. **监听回复**: 从 Message Bus 获取 `OutboundMessage`
5. **发送**: 通过平台 API 发送回复

**示例**: Telegram 渠道

```python
class TelegramChannel(BaseChannel):
    """Telegram 适配器"""

    async def start(self):
        """启动消息轮询"""
        while True:
            updates = await telegram_api.get_updates()
            for update in updates:
                msg = InboundMessage(
                    channel='telegram',
                    user_id=update.user_id,
                    text=update.message,
                    metadata={'chat_id': update.chat_id}
                )
                await self.bus.dispatch_inbound(msg)

    async def send(self, outbound_msg):
        """发送消息"""
        await telegram_api.send_message(
            chat_id=outbound_msg.metadata['chat_id'],
            text=outbound_msg.text
        )
```

**关键设计**:
- **无 if-elif 链**: 所有配置都通过注册表驱动

- **模型自动匹配**: 根据关键词选择最合适的提供商

**工作流**:

```
用户设置 model="claude-opus"
        │

        │
        ├─ 匹配 "claude" → openrouter/anthropic/...
        │
        ▼

        │
        ▼
```

### 6. Context Builder (`agent/context.py`)

**职责**: 构建最终发送给 LLM 的提示词

```python
class ContextBuilder:
    """
    组装完整的 LLM 提示词，包括:
    - 系统指令
    - 对话历史
    - 用户记忆
    - 技能描述
    - 工具定义
    """

    def build(self, user_id, new_message):
        prompt = []

        # 1. 系统提示词
        prompt.append(self.system_prompt)

        # 2. 用户记忆
        memory = self.memory_store.load(user_id)
        if memory:
            prompt.append(f"用户记忆:\n{memory}")

        # 3. 对话历史（最近 10 条）
        history = self.session_manager.load_history(user_id)
        for msg in history[-10:]:
            prompt.append(f"{msg.role}: {msg.text}")

        # 4. 新消息
        prompt.append(f"用户: {new_message}")

        # 5. 工具定义（JSON schema）
        tools = self.tool_registry.schema()

        return {
            "messages": prompt,
            "tools": tools,
        }
```

### 8. Periodic Services

#### 8.1 Heartbeat Service (`heartbeat/`)

**职责**: 定期检查并执行周期任务

```
[Gateway 运行]
        │
        ├─ 每 30 分钟检查一次
        │
        ▼
   检查 ~/.nanobot/workspace/HEARTBEAT.md
        │
        ├─ 读取任务列表 (Markdown checkbox)
        │
        ├─ 对每个未完成的任务:
        │  └─ 生成消息给 Agent Loop
        │  └─ 由 Agent 执行（可能调用工具）
        │  └─ 发送结果到最近活跃的渠道
        │
        └─ 更新 HEARTBEAT.md (mark as complete)

例如任务:
- [ ] 检查天气预报并发送摘要
- [ ] 扫描邮件找急件
```

## 设计原则

### 1. **极简主义**
- 核心流程集中在 Agent Loop
- 无复杂的状态机
- 无嵌套的抽象

### 2. **插件化架构**
- 通过 Registry 模式添加功能
- 无 if-elif 链
- 配置驱动而非代码驱动

### 3. **异步优先**
- 所有 I/O 都是异步的
- 支持并发处理多个用户
- 事件驱动 (Event-Driven)

### 4. **松耦合**
- 渠道和核心通过 Message Bus 解耦
- 工具独立实现
- 提供商通过注册表管理

### 5. **可研究性**
- 代码易读且易修改
- 清晰的模块边界
- 文档齐全

## 总结

nanobot 的架构遵循**极简、模块化、事件驱动**的原则，通过以下方式实现 99% 的代码精简：

1. **单一职责**: 每个模块专注一件事
2. **无冗余**: 通过 Registry 模式消除重复代码
3. **配置驱动**: 用配置代替代码
4. **事件队列**: 解耦各层组件
5. **异步 I/O**: 高效并发处理

这使得 nanobot 不仅轻量级，还非常易于理解、修改和扩展。
